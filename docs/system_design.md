# Системный дизайн: AI Career Navigator

## Ключевые архитектурные решения

1. **Оркестрация через агента:** Ядром системы является LLM-агент, который сам решает, когда использовать инструменты, а когда ответить напрямую. Это обеспечивает гибкость и адаптивность диалога.
2. **Разделение инструментов:** Векторный поиск (база знаний) и веб-поиск (актуальные данные) реализованы как отдельные, четко специфицированные инструменты, которые агент вызывает по мере необходимости.
3. **Stateful-сессии:** Состояние диалога управляется на уровне Telegram-модуля и передается в LLM через историю сообщений, что упрощает архитектуру на данном этапе.

## Список модулей и их роли

1. **Telegram Bot:**
   * **Роль:** Точка входа, оркестратор. Принимает сообщения, управляет сессиями, вызывает агента, отправляет ответы.
2. **Agent:**
   * **Роль:** Ядро принятия решений. На основе системного промпта и истории выбирает действие (ответ/вызов инструмента).
   * **Технологии:** YandexGPT API (или другая LLM).
3. **Retriever:**
   * **Роль:** Инструмент для семантического поиска профессий.
   * **Технологии:** FAISS, Yandex Embeddings.
4. **Web Search Tool:**
   * **Роль:** Инструмент для получения актуальной информации о рынке труда, зарплатах, трендах.
   * **Технологии:** API поисковых систем (например, Tavily).
5. **Memory/State:**
   * **Роль:** Хранение контекста беседы.
   * **Реализация:** In-memory словарь в Telegram-модуле, ключ — `user_id`.

## Основной workflow выполнения задачи

1. **Получение запроса:** Telegram Bot получает сообщение от пользователя.
2. **Загрузка контекста:** Бот загружает историю сообщений для данного `user_id`.
3. **Вызов агента:** Бот формирует промпт (системный + история + новое сообщение) и отправляет запрос к `model/` API.
4. **Цикл рассуждения агента:**
   * LLM анализирует запрос.
   * Если LLM решает, что нужны инструменты, она возвращает специальный формат (например, JSON) с названием инструмента и параметрами.
   * Бот выполняет соответствующий инструмент (поиск в FAISS, веб-поиск).
   * Результат выполнения инструмента отправляется обратно LLM для следующего шага.
   * Цикл повторяется, пока LLM не сформирует финальный ответ.
5. **Формирование ответа:** Получив финальный ответ от агента, бот отправляет его пользователю.
6. **Обновление истории:** Бот добавляет пару (запрос пользователя, ответ агента) в историю сессии.

## Описание retrieval-контура

* **Источники:** Данные из `data/summary_hh_professions_comparison.json` (агрегированные данные о профессиях с hh.ru).
* **Индекс:** FAISS, построенный на эмбеддингах текстовых описаний профессий.
* **Поиск:** Семантический (косинусное сходство). Запрос пользователя или запрос от агента эмбедится и ищется в индексе.
* **Ранжирование:** Возвращаются топ-K (K=5) наиболее семантически близких профессий.

## Описание tool/API-интеграций

* **Retriever:** Внутренний Python-вызов `search_professions` с параметром `query`. Возвращает список профессий с описаниями.
* **Web Search:** Внешний API. На вход принимает строку запроса (например, "зарплата data scientist 2026"). Возвращает структурированные результаты (заголовки, ссылки, сниппеты).
* **LLM API:** Эндпоинт `/start_talk/` (существующий). Может быть расширен для поддержки формата вызовов инструментов (function calling).

## Основные failure modes, fallback и guardrails

* **Таймаут LLM API:** Если ответ не получен за 10 секунд, бот возвращает сообщение о технических неполадках и предлагает повторить запрос позже.
* **Ошибка Retriever:** Если FAISS индекс не найден или поиск не дал результатов, агент получает специальное сообщение: "По вашему запросу в базе профессий ничего не найдено." Агент должен ответить, что не может подобрать профессию и предложить уточнить запрос.
* **Ошибка Web Search:** Если поиск не удался, агент должен сообщить, что не смог найти актуальную информацию, и основывать ответ только на данных из индекса.
* **Бесконечный цикл агента:** Установлен жесткий лимит на количество итераций "рассуждение → вызов инструмента" (например, 5). При превышении лимита бот прерывает цикл и отправляет пользователю сообщение о сложности запроса и предложение обратиться к консультанту.

## Технические и операционные ограничения

* **Latency (p95):** < 15 сек.
* **Cost:** < $1 за сессию.
* **Reliability:** Система должна корректно обрабатывать ошибки всех внешних сервисов (LLM, веб-поиск).
